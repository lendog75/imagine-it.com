Case Study: ToolWatch Shinobi Project

Need/Problem

ToolWatch's flagship applications is a very dated winforms app.

  • Used a very complex and unstable merge replication system to allow users to work offline.
  • Larger organizations suffered from poor performance
  • All users have been plagued with unreliable behavior
  • The app is tightly coupled. It all begins with the forms and grids, the business and data layers are built upon grid objects, not POCO's
  • The nature of the replication system makes any changes very difficult and extremely risky. There have been 2 outages lasting 2 weeks or more in the last year.
  • Deployment uses click-once, and different clients are on different versions.
  • Based on a single SQL Server database

Requirements

  • The new app should be web based
  • It should be connected
  • Use a newly developed Restful API
  • UX is to be considered a priority
  • Must work on modern browsers
  • Must be backward compatible with the legacy application
  • Should use Google, Amazon, E bay and other commercial sites as inspiration
  • Initial Minimum features include: Search, Tools, Employees, Locations, Move, and Authentication

Solution

It was decided we would develop A new application based on HTML5, CSS3, AngularJS, and Bootstrap. The new framework would use the newly developed enterprise API (REST). This will reduce the load on the single SQL server database (due to constantly cued merges)

  • HTML5, CSS3, and Bootstrap will allow us to provide a fresher more interactive user experience.
  • Angular allows the UI logic to be shifted client side, while the API will ultimately be responsible for ensuring all business logic/rules are implemented


Module: Core

The core module is designed to be exactly that, it is the core, all other modules depend on it. It includes features such as Authentication, Navigation, and in-app Notifications. It is required to be included in each sub-app.

Authentication is done against the Security API using email, password, and accesskey (assigned to each application). Upon successful authentication an authorization token is returned to the client. This token is to be sent along in all subsequent requests, identifying the user. If at any point client receives a status code 401, the app then requires the user to login again.

Navigation is rather simple, the core however keeps track of which module the user is in making it as active in the navigation.

The final feature to discuss is Notifications. Shinobi was designed to be connected from the beginning, one feature that was paramount was notifications. At design time, there were really two types of notifications. The first is what we call in-app notifications. These notifications are a way for the app to communicate important information to the user as it happens. Examples include an action being canceled, records being added to a move, or a move being submitted to ToolWatch and so on. Currently all in-app notifications are centered around the Move module (discussed later on).

The second type of notification, comes from the server, an example of this is the server notifying the client that their move has been processed. This second type of notification becomes quite critical as the team moves further along the road map, moving into distributed systems. Once work becomes asynchronous, usually through queuing then there will be a bit of a disconnect between users requesting and action and the completion of that action (the is often referred to as “eventual consistency”). Implementing such systems are critical to taking the next step in scalability. Ultimately, server notifications will help users know when there request has been filled. Lateron, the roadmap calls for socket.io to further improve the user experience and feeling of being connected to ToolWatch systems.

Module: Tool

The app at its core is an asset tracking/management system, and the most important assets to the system are tools. So it is no shock that the tool module gets the most attention.

Feature: Tool Search

The tool search is probably the most exciting part of the system. It starts with an Google style auto-suggest. This is backed up by a lightning fast Search Api (part of the Enterprise API). Each keystroke calls the API and the results are sorted by relivance, returning upto 100 results. Each result represents a model or category each representing many individual tools. Honestly, I would love to say more, but is so unbelievably simple!

After selection the model or category, you want it leads to a results page. The Amazon style results page detailing what was selected and how many results were found. below that there are two panes, on the left are related filters including manufacturer, assignment, and status. Selecting any of these options immediatly filters the result set. On the right is a list of results including pagination, sorting, and actions that can be taken on each item.

Feature: Tool Detail

One of the actions that can be taken from the search results is getting the detail of a specific tool. Here you can see critical info such as the model, description, manufacturer, notes, and the individual assignments for the tool

     
     

Module: Employee

Feature: Employee Detail

This is a rather simple module, as with tools it starts with the search. After selecting a specific employee you are taken to the employee detail where (like tool detail) you are presented relivant details on the employee including name, contact info, and tools assigned to that person.



Module: Location

Feature: Location Detail

As with employee this is a simple module, and again you can find locations using the search. After selecting a specific location you are taken to the location detail where (like tool detail) you are presented relivant details on the location including name, address info, and tools assigned to that location.

Unlink employees, locations also allow for a way to see the complete list, in a Google style result set. Filtering is included, limiting the list as you type.
